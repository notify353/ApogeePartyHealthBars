# Publish Release

Treat explicit invocation as authorization for release preflight, preparation, and documented non-public setup, never as approval to make a release public or deploy it; read the repository's release instructions, and if the production mechanism or destinations are undocumented or materially ambiguous, stop and ask rather than inventing a workflow; preserve unrelated work and stop if it prevents required clean-state checks rather than discarding, hiding, or absorbing it; confirm the exact release identifier and destinations, then determine the current stage from repository state, preparation commits, release identifiers, CI, and publication records; continue from the first incomplete safe step without repetition: prepare only when needed, reuse existing preparation work, resume pending CI, never recreate an existing public identifier, and verify and report if publication is already complete; confirm prerequisites, a clean synchronized release source, and landed changes, then run only the documented preparation and validation process; immediately before the first command or API call that can publish artifacts, push a production tag, or deploy, present required manual checks and obtain fresh explicit approval naming the exact release and destinations—the initial invocation and earlier, silent, or ambiguous approval do not count; after approval, revalidate state, use only the documented publisher, monitor it to completion, and verify versions, artifacts, checksums, and destinations when applicable; never bypass release automation, expose secrets, overwrite or reuse a published identifier, or improvise a repair after partial publication; preserve state and report failures, then finish with the release identifier, commits or tags, workflow result, artifacts, URLs, and remaining checks.
