# Land Changes

Treat explicit invocation as authorization only for version-control and code-review operations needed to land the intended change, not for a release or deployment; read repository instructions, preserve unrelated work, discover the default branch and established landing workflow, and determine the current stage from the working tree, branch commits, remotes, review, and required checks; continue from the first incomplete step: when intended changes are uncommitted, inspect the relevant diff, run canonical validation, create a short-lived branch when required, stage only intended paths, and make clear logical commits; when changes are already committed, validate the relevant commit range; then push the existing branch, create or update one review, wait for required checks, and merge using the established strategy; if already merged, synchronize the default branch and report completion, and if nothing exists to land, report a no-op; ask rather than invent when the workflow, destination, or merge strategy is materially ambiguous, and stop on validation failures, unclear scope, conflicts requiring judgment, or missing authentication; never create empty or duplicate work, rewrite published history, force-push, tag, release, upload artifacts, deploy, or expose secrets; clean up branches only when safe and customary, and report commits, review URL, checks, merge result, final repository state, and remaining manual checks.
